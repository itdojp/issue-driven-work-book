# 第1章：チケット駆動の前提（成果物・品質・期限・責任分界）

## この章で学ぶこと

- 口頭依存/属人化を減らすために、情報をチケットに集約する
- 成果物（何が出来たら完了か）を先に定義する
- 品質・期限・責任分界を明文化する

## 成果物（書けるようになるもの）

- Issue に“目的/スコープ/受け入れ条件”を書ける
- 関係者の責任分界（誰が判断/誰が実行）を書ける

## 本文

チケット駆動の価値は、作業の可視化だけではなく、合意形成と再現性にある。

### 最小の構造

- 背景（なぜ今必要か）
- 目的（どうなれば成功か）
- スコープ/非スコープ
- 受け入れ条件（検証可能な形）
- 期限/優先度
- 責任分界（判断者/実行者）

この構造がないと、レビューや引き継ぎで“口頭の補足”が増え、事故や手戻りにつながる。

## 具体例（悪い例→良い例）

### 悪い例

```md
タイトル: 画面を直す
本文: なんか使いにくいので改善してください
```

### 良い例

```md
タイトル: 入力フォームの誤操作を減らす
背景: 入力ミスによる差戻しが多い
目的: エラー発生率を下げる
スコープ: 必須項目のバリデーション/文言
非スコープ: 画面全体のデザイン刷新
受け入れ条件: Given… When… Then…
責任分界: 判断=PM、実行=開発、確認=CS
```

## チェックリスト

- [ ] 背景/目的がある
- [ ] スコープ/非スコープがある
- [ ] 受け入れ条件が検証可能
- [ ] 責任分界（判断/実行/確認）が書かれている

## まとめ

- テンプレと受け入れ条件で“期待する結果”を固定し、手戻りを減らす
- 記録（ログ/証跡）を残し、レビュー可能な状態にする
